# Dependency Management

Every third-party dependency is code you didn't write but are responsible for running, patching, and
trusting anyway. Dependency management is the discipline of making that an ongoing, low-drama
process instead of a periodic fire drill.

## Vetting new dependencies

Before adding a new dependency, weigh it against writing the functionality yourself or using
something already in the codebase:

- **Maintenance health** — is the project actively maintained (recent releases, responded-to
  issues), or effectively abandoned? An abandoned dependency accrues unpatched vulnerabilities and
  eventually blocks upgrades of everything around it.
- **License compatibility** — does its license permit the intended use, including
  commercial/proprietary use if applicable? Check this before integration, not after legal notices
  it during a release review.
- **Transitive dependency footprint** — what does this dependency pull in underneath it? A single
  package can bring in dozens of transitive dependencies, each an additional piece of the
  vulnerability and maintenance surface you're now responsible for.
- **Actual necessity** — does this dependency solve a real, current problem, or a convenience that
  could be a few dozen lines of code you'd fully understand and control? A small, well-scoped
  dependency for a genuinely hard problem (cryptography, date/timezone handling) is usually worth
  it; a large dependency to avoid writing ten lines of straightforward logic often isn't — see
  [`yagni-principle.md`](./yagni-principle.md) applied to external code, not just internal
  abstraction.
- **Security track record** — has this project had vulnerabilities, and how quickly were they
  patched when found? A history of fast, responsible disclosure handling is a positive signal; a
  history of unpatched known issues is a strong negative one.

Record the decision to adopt a significant new dependency (one with wide usage across the codebase,
or one in a security- or data-sensitive path) the same way as any other consequential decision — see
[`adr-guide.md`](./adr-guide.md) — so a future engineer knows why it was chosen over the
alternatives.

## Update cadence

- **Patch and minor updates**: apply on a regular, largely automated cadence (weekly or biweekly is
  a reasonable default) rather than batching them up. Small, frequent updates are individually
  low-risk and easy to bisect if something breaks; a batch of six months of accumulated updates
  applied at once is not.
- **Major updates**: apply deliberately, on their own schedule, with the breaking-change review the
  update's own changelog calls for. Don't let major updates accumulate indefinitely either — the
  longer a major version lags behind, the larger and riskier the eventual jump becomes, and the more
  likely the vendor has stopped supporting the old version entirely.
- **Security patches**: apply out-of-band, as soon as a fix is available and verified, independent
  of the regular cadence — see the vulnerability scanning section below for how these get surfaced.
- Automate what can be automated (dependency update PRs generated by tooling, auto-merge for
  patch-level updates that pass full CI) so the update cadence doesn't depend on someone remembering
  to do it manually.

## Lockfiles

- **Commit lockfiles to version control.** A lockfile pins the exact resolved version of every
  dependency (direct and transitive), so a build today and a build next month from the same commit
  produce identical dependency trees. Without one, "it worked when I built it" stops being a
  guarantee.
- **Update lockfiles deliberately, as part of a reviewed change**, not silently regenerated as a
  side effect of an unrelated command. A lockfile diff should be reviewable — a reviewer should be
  able to see which dependencies moved and to what version as part of the PR.
- **Never hand-edit a lockfile.** Let the tooling regenerate it from the manifest; hand edits risk
  producing an inconsistent state the tooling itself can't reproduce or verify.

## Vulnerability scanning cadence

- Run automated dependency vulnerability scanning on every build in CI, not as a periodic manual
  audit — a scan that only happens quarterly leaves a window of unknown length where a known,
  patchable vulnerability sits in production.
- Triage findings by actual exploitability and exposure, not by severity score alone — a critical
  vulnerability in a transitive dependency that's never invoked on any reachable code path is a
  different priority than a critical vulnerability in a directly-used, internet-facing component.
  Don't let this triage become an excuse for inaction, though: undetermined exploitability should
  default to prompt patching, not indefinite deferral.
- Set an explicit maximum time-to-patch for vulnerabilities above a severity threshold (e.g.,
  critical within days, not sprints), and track exceptions the same way any other accepted risk is
  tracked — see [`risk-assessment.md`](./risk-assessment.md).

## Deprecation handling

- When a dependency you rely on announces deprecation or end-of-life, treat that as a tracked piece
  of technical debt immediately (see [`technical-debt.md`](./technical-debt.md)), not something to
  address only once it actually breaks.
- Plan the migration to a replacement before the deprecated dependency's support window actually
  closes — migrating under time pressure after support has already ended is a materially worse
  position than migrating on a planned timeline.
- When deprecating your own internal dependency that other teams consume, apply the same discipline
  you'd want from an external vendor: advance notice, a documented migration path, and a defined
  support window — see [`api-design-guide.md`](./api-design-guide.md).
